# Vibe Dev：文档驱动的 AI 编程方法论

## 核心理念

AI 像一个"很能干但时不时犯傻的新人"。你的角色是**架构师和决策者**，不是监工。

- AI 负责高速输出代码
- 你负责判断、决策、定位根因
- **方向盘必须握在你手里**

---

## 开发流程

### 第零阶段：检查项目文档状态

**重要**：每次对话开始时，先检查项目是否已有文档：

1. 检查 `docs/README.md` 是否存在
2. 检查 `docs/ARCHITECTURE.md` 是否存在
3. 检查 `docs/PROJECT_STRUCTURE.md` 是否存在（项目结构详解）
4. 检查主要 src 文件夹是否有 `FOLDER.md`

**如果项目已有完整文档**：
```
已检测到项目已有完整文档结构：
- docs/README.md
- docs/ARCHITECTURE.md
- docs/PROJECT_STRUCTURE.md
- [各 FOLDER.md 文件列表]

我将先阅读这些文档了解项目背景，然后再开始工作。
```

**如果项目未初始化文档**：
```
检测到项目尚未初始化文档结构。
请选择：
A) 初始化文档（推荐）- 为项目生成完整的文档结构
B) 跳过初始化 - 直接开始工作，不建立文档

请选择 [A/B]：
```

---

### 初始化文档（适合老项目）

当用户选择 **A) 初始化文档** 时，执行以下流程：

**Step 1：探索项目结构**
- 先列出项目根目录下的所有文件和文件夹
- 了解项目类型（.NET/Java/Node.js/Python 等）
- 根据项目类型探索对应目录结构（如 src/、lib/、app/、packages/ 等）
- 了解技术栈（查看 .csproj、package.json、requirements.txt 等）
- 了解项目职责（查看 README 或 main 文件）

**Step 2：生成根目录文档**
```markdown
# 项目文档

**更新规则**：任何功能、架构、写法更新，必须在工作结束后同步更新相关目录的子文档。

遇到问题时，先查看本目录下的文档，了解上下文后再开始工作。

## 文档索引

| 文档 | 用途 |
|------|------|
| [ARCHITECTURE.md](./ARCHITECTURE.md) | 系统架构总览 |
| [PROJECT_STRUCTURE.md](./PROJECT_STRUCTURE.md) | 项目结构详解（快速定位功能） |
```

**Step 3：生成架构文档** (`docs/ARCHITECTURE.md`)
- 核心模块划分
- 数据流转关系
- 技术选型
- 更新日志

**Step 4：生成项目结构详解** (`docs/PROJECT_STRUCTURE.md`)
- 完整目录结构
- 所有模块的职责说明
- 快速定位指南
- 新增功能检查清单

**Step 5：生成各文件夹 FOLDER.md**
- 根据实际项目结构，为主要代码文件夹生成 `FOLDER.md`
- 三行以内，包含职责和文件列表
- 示例：`src/*/FOLDER.md` 或 `app/*/FOLDER.md` 或根目录下的各模块文件夹

**Step 6：生成每日变更日志** (`docs/superpowers/DAILY.md`)
- 记录每次变更

### 第一阶段：全局蓝图（新项目一次性）

**目标**：在脑中构思整个系统的架构，输出到一份完整的方案文档里。

**步骤**：
1. 明确系统要解决的核心问题
2. 划分核心模块
3. 确定数据流转和关键依赖关系
4. 输出到 `docs/ARCHITECTURE.md`

**输出文件**：`docs/ARCHITECTURE.md`

---

### 第二阶段：建立分形文档结构

**目标**：建立自指的文档体系，让 AI 在任何时候都能回到之前的状态继续工作。

#### 三层文档结构

**第1层：根目录文档**
```
docs/
├── README.md          # 根文档，声明更新机制
├── ARCHITECTURE.md    # 系统架构总览
├── PROJECT_STRUCTURE.md # 项目结构详解（快速定位）
└── superpowers/
    └── DAILY.md       # 每日变更日志
```

**第2层：文件夹极简架构说明**（每文件夹一个 `FOLDER.md`，三行以内）
```markdown
# [文件夹名] 架构说明

**职责**：[一句话描述]
**包含**：[文件名] - [功能], [文件名] - [功能]

> ⚠️ 一旦本文件夹有变化，请更新本文档
```

**第3层：代码文件头部三行注释**
```typescript
// input:  [依赖外部的什么]
// output: [对外提供什么]
// pos:    [在系统局部中的地位]
// ⚠️ 一旦此文件被更新，务必更新头部注释及所属文件夹的 FOLDER.md
```

**自指机制**：局部影响整体，整体影响局部。修改任何文件时，自动触发一连串的文档同步。

---

### 第三阶段：逐模块开发（核心循环）

每个功能模块都按以下步骤执行：

#### Step 1：方案设计（方案先行）

**不要直接写代码**。先让 AI 输出技术实现方案，你 review 和微调。

Prompt 模板：
```
在实现 [模块名] 之前，请先输出该模块的技术实现方案，包括：
1. 数据模型设计（表结构或类型定义）
2. 核心接口（函数名、参数、返回值）
3. 与其他模块的依赖关系
4. 关键实现细节
5. 可能的坑

等我确认后再开始写代码。
```

**你（人类）的职责**：
- 审查方案是否合理
- 调整权限边界设计
- 确认技术选型
- 补充遗漏的边界情况

#### Step 2：小任务拆分

将模块拆分成**若干个可独立完成的小任务**。

每个小任务包含：
- ✅ 明确的目标
- ✅ 技术方案
- ✅ 实现步骤
- ✅ 验证方式

#### Step 3：逐个小任务实现

**顺序**：基础层 → 业务逻辑 → UI 层

每完成一个小任务后，**立即**：
1. 更新代码文件的头部注释
2. 更新所在文件夹的 `FOLDER.md`
3. 运行验证（单元测试、demo 页面、或手动测试）

**Prompt 模板**（验证后提醒 AI 同步文档）：
```
验证通过。现在请：
1. 更新 [文件名] 的头部注释（如果实现有变化）
2. 更新 docs/[文件夹]/FOLDER.md（如果接口有变化）
3. 如有跨文件夹依赖，同步更新相关文档
```

#### Step 4：端到端集成

所有模块完成后，进行端到端测试。

---

### 第四阶段：调试（人类掌舵的关键时刻）

**最重要的原则**：当同一个问题修改超过 2-3 次还没解决，**立即停止**，这表明模型陷入了错误框架。

#### 调试步骤

**Step 1：识别危险信号**
- 模型在同一个问题上修改超过 2-3 次
- 按了葫芦起了瓢（改了 A 出 B，改了 B 出 C）
- 代码越改越乱

**Step 2：人类亲自定位根因**
- 看报错信息，但不要只看表面
- 用日志、断点、代码追踪找到真正原因
- 问自己：问题发生在哪个模块？为什么发生？根本原因是什么？

**Step 3：明确告诉模型根因**

❌ 不要说：`这里有 bug，请修复`
✅ 要说：`你之前的假设是错的。真正的问题在于：[具体描述根本原因，包括为什么]。请基于这个理解，重新实现。`

**Step 4：让模型基于正确理解重新生成**

---

### 第五阶段：迭代

| 场景 | 入口 |
|------|------|
| 新功能 | 回到 Step 1（方案设计），在系统背景中注明现有技术栈 |
| 性能/UX 问题 | 进入调试模式，描述问题 + 贴相关代码 |
| 代码混乱 | 重新设计模块边界，再开始添加功能 |

## 人机协作角色表

| 环节 | 你的职责 | AI 的职责 |
|------|---------|----------|
| 方案设计 | 架构决策、权限边界、技术选型 | 方案评审、可行性分析、细节补充 |
| 代码实现 | 方案审查、代码审查、关键排错 | 体力活（增删改查、接口文档、字段同步） |
| 问题定位 | **根因分析、问题诊断** | 基于指导进行修复 |
| 测试验证 | 测试设计、边界场景补充 | 测试脚本生成、demo 页面 |

## AI 常见坑及应对

### 1. API 幻觉
AI 会虚构不存在的 API、库函数。
**应对**：在 prompt 中强调"使用官方文档中存在的 API"，必要時查官方文档核实。

### 2. 错误假设上反复打补丁
AI 在错误理解基础上反复修改，越改越乱。
**应对**：亲自定位根因，明确告诉 AI 错误的假设是什么。

### 3. 过度设计
AI 生成过多设计模式、工厂函数、装饰器。
**应对**：代码审查时直接删掉，保持代码简洁。

### 4. 边界场景遗漏
AI 只实现 happy path，空值/异常/并发处理不周。
**应对**：在 prompt 中提前列举需要处理的边界场景，或在测试阶段补充。

## Prompt 模板库

### 开始新项目
```
我要开始一个新项目：[项目描述]
请先帮我输出项目架构文档，包括：
1. 核心模块划分
2. 数据流转关系
3. 技术选型建议

等我确认后再开始建立文档结构和代码。
```

### 添加新功能
```
我要为 [现有模块] 添加 [新功能]。
请先输出技术实现方案，等我确认后再写代码。
```

### 调试请求
```
遇到了问题：
- 现象：[描述]
- 预期：[期望行为]
- 实际：[实际行为]

我已经尝试过：[尝试的方案]

在让我修复之前，请先分析可能的原因。我会告诉你根因后再修复。
```

### 更新后同步文档
```
[模块名] 已完成。请同步更新：
1. 代码头部注释（如果接口有变化）
2. docs/[文件夹]/FOLDER.md
3. docs/ARCHITECTURE.md（如有重大变更）
4. docs/PROJECT_STRUCTURE.md（如有新增模块）
```

### 初始化老项目文档
```
这是一个老项目，还没有初始化过文档。请帮我初始化项目的文档结构。
```

> 注意：初始化时会先探索项目的实际目录结构，不预先假设特定结构（如 src/），然后根据实际情况生成对应的文档。

## 效率目标

如果严格执行这套流程，真正需要你手写的内容应该 < 5%：
- 文案措辞微调
- 根因分析和问题诊断
- 边界情况处理决策
- 代码审查和架构调整

AI 负责剩下的 95%：体力活、重复劳动、高速代码生成。
